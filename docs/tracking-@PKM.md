### 几级跳

1. **锚图教会一个方法**：长 moc 的锚，终点是"锚句 + 一张表"，不是图。图（层级+桥+箭头）帮你边画边想，想清楚就塌缩成表（图底翻转 → 因果链 → 砍冗余 → 塌成表）。AI 出 grounded 原料，压缩——画/写——必须自己过手（→ memory `feedback_anchor_figure_method`）。
2. **接入 [moc-@INTJ人生指南](moc-@INTJ人生指南.md) 是转折**：浮现两条正交轴——**认知链路 = Ni 轴**（认知怎么运作·镜子），**INTJ moc/ref = Te 轴**（我该怎么做·操作手册）。锚图的"行动层"必须从 Te 轴取材；从 v2（Ni 轴）取，只会得到"认知提升"而非"行动"。
3. **§4 是两轴接缝**：INTJ moc §4 干的就是"把一个 Ni 描述模型翻成对我的 Te 操作"。已找到 4 实例（认知链路→能力圈 / 多巴胺→内部积分卡 / 君主论→自我提升 / §4 本身）。隐约觉得"**X 和我有什么联系？**"就是整个库一直在做、却没命名的核心动作。
4. **闭环浮现**：模型 → §4 翻译 → 多巴胺评估 → 行动 → **tracking（流水账）→ 反哺模型**。tracking 是从"我"回到"模型"的**回程弧**——正是 _todo 延后的 A2，现在有了意义。
5. **正在长出第 2 篇"归纳优先框架"**：这些挖矿汇成一篇"书接上回 moc-v2"，大概率就是 v2 分类问题等的 **n=2 样本**。

### 协作层收获

人机协作 = **Ni-Te-Fi-Ti 循环**（我出 Ni+Fi，AI 出 Te+Ti）。两陷阱：Ni-Ti（笔记落地 ≠ 理解落地，要留消化时间）、Ni-Fi（茧房，AI 顺我偏好走，须主动反问"我漏了什么 / 这是真问题吗"）（→ memory `feedback_ni_te_fi_ti_loop`）。

## 2026-05-06：Canvas 在 PKM 体系中的位置

> **来源声明**：本节由 Claude 代写，基于 2026-05-06 与作者关于 Obsidian Canvas 第一次实战的讨论整理。作者负责 review 与最终判断。

### 引言

作者 5 月 5 日决定先沉淀已有的 !q- / card / moc，开始用 Obsidian Canvas 做可视化 review。第一张画布建好后，作者发现一个重要的认知：**Canvas 不只是看笔记的工具，它是 PKM 体系里一个之前缺失的层**。

---

### 一、Canvas 的真正位置

之前作者把笔记分成几类：book / ref（输入）、card / toolkit（加工）、tracking（过程）、moc（导航）、!q-（开放问题）。这些都是"已经成形的"内容。

Canvas 补的是另一种东西——**正在成形中的关系**。它处在内容节点之间的关系层，让作者可以**先把关系画出来，再决定是否把它写成稳定的 card 或 moc**。

可以这样描述各类笔记的"结晶程度"：

- 输入态：book / ref
- 探索态：**Canvas**（关系流动，可随时改）
- 半结晶态：!q-（问题清楚但答案未定）
- 已结晶态：card-@（单概念稳定）
- 已合成态：toolkit-@
- 已导航态：moc-@（文字版）

Canvas 是上游探索工具，其他类型是下游沉淀成果。

---

### 二、Canvas 不能完全替代文字笔记

讨论中作者的两点澄清：

**1. card 仍然必要**

当一个概念已经被反复确认稳定（在多张 Canvas 上重复出现、关系清楚），它就该有一个稳定的文字载体——就是 card。Canvas 是临时视图会过时，card 是长期参考。

**2. 文字版 moc 仍然必要**

Canvas 上画的关系图属于"我现在的解读"。但 moc 是问题导航地图，需要稳定的文字陈述驱动问题、引导到具体笔记。Canvas 是 moc 的可视化辅助，**不是替代**。

---

### 三、Canvas 可能改变协作模式（仅作可能性，不固化）

之前作者和 Claude 的协作是：Claude 写半成品 → 作者填 TODO 沉淀。

5 月 6 日讨论中浮现一种新的可能性：**作者在 Canvas 上画关系（找到框架）→ Claude 写完整笔记 → 作者 review**。

## 2026-05-05：关于"沉淀"和"加机制"的两难

> **来源声明**：本节由 Claude 代写，基于 2026-05-05 与作者关于 _todo P1（A1+A2 tracking 方法论）的讨论整理。作者负责 review 与最终判断。

### 引言

作者准备开始做 _todo §一 P1 的 A1+A2（tracking 方法论 + 行动→提炼闭环）。和 Claude 讨论一轮后，方案从"给 tracking 加 metadata + 每月 audit + 概念激活机制"逐步收缩，最后变成几乎不动现有体系。这个**收缩本身**就是这次的核心 lesson。

---

### 一、几条核心 lessons

**1. tracking 其实就是特定类别的 inbox**

讨论中作者突然意识到：inbox 和 tracking 用的是同一个机制——最小阻力记录 + AI 后置判定。区别只在 inbox 是临时态（目标是清空），tracking 是持续态（目标是累积）。

这个观察让整个体系更连贯——所有"输入"类内容都遵循"先记录后结构化"的统一原则。

**2. AI 结构化的产出不能污染流水账**

作者担心：如果给 tracking 每条记录都加 metadata 字段（applied / verdict / update），后面回看的时候要绕过这些结构化字段才能看流水，违反最小阻力原则。

解法是把 AI 反馈做成 tracking 文件末尾的独立索引区，和流水账用一道分隔线隔开。但这一轮先不动——等第一次真跑 audit 时再加。

**3. 不要在用户消化期间加新机制**

作者最关键的一条 tension：这几天高强度对话产出了大量新建的 !q- / card / moc / toolkit，但还没时间沉淀。再加 audit 机制会过载。

Claude 之前默认 "AI 协作 = 帮用户做更多事"。这次作者反过来指出：**复利的引擎不是 AI 多干活，是 AI 知道什么时候让用户独自待一会**。

这条改变了后面所有具体决定：audit 改成触发式（用户说"帮我 audit"才跑）、代写笔记改成半成品（只填核心字段，其他留 TODO）、tracking 文件结构延后到真需要时再动。

**4. Claude 代写的笔记应当留半成品状态**

作者反思：哪怕全程 review 过 Claude 代写的内容，"看一遍就过"的浅层 review 让笔记停留在 Claude 的认知里，没真进作者的认知。

解法：未来 Claude 代写 !q- / card / moc / toolkit 时，只填核心字段（锚点句 + 主张力），其他字段留 TODO。让作者在沉淀过程中自己填——填的过程就是吸收的过程。

这条作为新 memory 已写入（feedback_audit_and_drafting）。

---

### 二、给自己的提醒

- 当前阶段不要再让 Claude 加新机制——已有 !q- / card / moc / toolkit / 候选池 / 工具决策自检 都还没用过几次
- 有"4 类 tracking 之外的散点" 时扔到 [_inbox](_inbox.md) §临时记录，等下一次主动触发 audit 时再分流
- 想跑 audit 时主动说一句话（"帮我 audit / 复盘 tracking / 看看 inbox 临时记录"）——AI 不会主动催
- 给自己几周时间沉淀，然后再决定下一步加什么 / 不加什么

---

### 三、本次最重要的一句

> 复利的真正引擎不是 AI 多干活，是 AI 知道什么时候让用户独自待一会。

## 2026-05-04：与 Claude 五十多回合对话之后的几条记录

> **来源声明**：本节由 Claude 代写，基于 2026-05-02 至 05-04 期间作者与 Claude 的对话整理。作者负责 review 与最终判断。

### 引言

最近三天作者和 Claude 高强度对话，把笔记体系做了一次大改造。改完之后，作者让 Claude 帮自己复盘一下"哪些经验以后能反复用"。下面是 Claude 整理的几条。

---

### 一、和 AI 对话这件事

**1. AI 最有用的时刻，不是回答问题，是帮你看清你真正想问的是什么**

作者在 5 月 1 日写了一篇笔记，七成篇幅都在纠结要不要加 Notion、Obsidian 怎么用。Claude 反过来指出：作者顺手写下的两条 ToDo（如何追踪 tracking、如何把 lessons 变成 checklist）才是真问题。前面所有的工具讨论，其实都是在绕开这两条。

作者后来自己说过一句话："答案的好坏取决于对问题的描述。"——这就是它的另一面。和 AI 对话最值钱的瞬间，不是它给你答案，是它帮你重新问。

**2. 写下一条原则，不等于以后会照着做**

5 月 1 日作者刚写完"不要再陷入鼓捣笔记软件陷阱"，紧接着在同一篇笔记里就开始认真考虑加 Notion。

写在卡片上的原则只是档案。它要真起作用，得绑定到一个具体动作（比如"我考虑加新工具的时候"）+ 一组强制问题。原则得变成"钩子"才能拦得下人。

这就是后来 [toolkit-@工具决策前的自检](toolkit-@工具决策前的自检.md) 的由来。

**3. Claude 代写的笔记和作者自己写的笔记，留下的痕迹不一样**

哪怕作者全程 review 过 Claude 写的内容，"看懂了"和"自己想出来的"，在脑子里留下的印记不是一回事。

所以约定：所有 Claude 代写的内容，开头都要写一句来源声明。这不是走形式，是防止几年后回看时，把 Claude 的推论错记成自己的洞察。

---

### 二、关于笔记体系本身

**4. 命名错了几个月，纠正只要半小时**

`atom-` 这个前缀用了几个月。它来自"原子化笔记"的概念，但作者实际用它来写开放问题，两件事根本不是一回事。每次写新笔记、向 AI 解释，都会有一点点别扭。

改成 `!q-` 用了 30 分钟。但消除的是这几个月里几十次回访时悄悄累积的摩擦。

教训：命名要尽早调对。错命名的代价是慢慢攒出来的，等你发现的时候已经付了很多。

**5. 二选一往往不够，要找第三档**

这次对话里反复出现这种情况：

- 不是"升级成 !q-"或"不升级"，而是"强升级 / 弱候选池 / 留 inbox"
- 不是"保留"或"删除"，而是"分流 / 归档 / 删除"
- 不是"重要"或"不重要"，而是艾森豪威尔的四象限

当一个判断只有两个选项的时候，多半是还没想透。强迫自己列第三个选项，往往会发现真实情况需要的就是那个第三档。

**6. cluster 是事后归纳的，不是事前过滤的**

作者一度担心：新笔记的主题不在现有 12 个 cluster 里怎么办？

Claude 提醒：作者 4 月 30 日那次分类本来就是先有 90 篇笔记、再归出 12 个 cluster——不是先建 cluster 再往里塞。所以处理孤立主题的方法是：先入库，把它链到最相近的 cluster；等同类第 3 篇出现，再自然形成新 cluster。

这条对所有"分类系统"都成立——分类是事后归纳工具，不是事前过滤器。

**7. 不要在跑通流程之前就优化**

作者一度想先把 metadata 设计、标题模板、card 数量都规划好再开始用。Claude 不同意：跑一篇真实的 ref 暴露出来的痛点，比预想的"可能问题"靠谱得多。

后来跑了 [ref-暖暖熊-如何读书读论文读教材读代码](ref-暖暖熊-如何读书读论文读教材读代码.md) 那篇，确实暴露出 7 个 README 不够用的卡顿点——其中有 4 个是事前完全想不到的。

教训：先用，再修。预防"可能的问题"几乎一定是浪费精力。

---

### 三、关于回访这件事

**8. 写下来不等于会被读到**

这次新建的几个东西都带回访机制：候选池有"升级信号"，`!q-` 有"待回看条件"，待读清单按 cluster 分区（这样撞到锚点 book 时容易触发）。

它们做的是同一件事：**没有触发条件的笔记 = 信息坟墓**。

知识管理的复利不来自"读了多少书"，而来自"留下了多少未来的自己能认出'这个值得回头看'的痕迹"。

---

### 四、给自己的提醒

- 候选池里挂的 3 条（二八规律 vs 简化论 / 大一统 / 形而上学）要真回看，否则就是新的 inbox
- [moc-@决策杠杆](moc-@决策杠杆.md)、[toolkit-@让系统1为我所用](toolkit-@让系统1为我所用.md) 这些升级，要被实战验证。一两个月后回看：完全没用到，就是精度通胀，该降级或归档
- `tracking-@` 方法论（A1+A2）是下一阶段重点，按 [_todo](_todo.md) P1 启动
- 别再陷入"先把所有规则定完再用"。已经 review 过 [toolkit-@工具决策前的自检](toolkit-@工具决策前的自检.md) 了，下次冒出"加新工具"冲动时去走一遍 5 问

### 三、最重要的几个判断

1. **笔记的组织轴是概念/主题/模型，不是书**——书只是 input。
2. **cluster ≠ moc ≠ card**——物理归簇 vs 问题导航 vs 单概念结晶；三层语义不同，混用即混乱。
3. **moc 必须有驱动问题**，否则只是个名字花哨的目录。
4. **Don't multi-purpose tools**——Typora 写、Obsidian review、card 是 concept 单元、moc 是 navigation 单元。每个工具 / 笔记类型负责一件事。
5. **AI 味儿的笔记不是垃圾，是未结晶的素材**——补 card 层后，它们成为 toolkit/card 的喂养源。
