# 锚点句

当我刚学完一个新概念 / 方法论时，我倾向于把它应用到所有场景以加深理解（**MVP 验证**）。但"刚学完 = 最容易想起"会让我系统性地高估它的适用范围（**可得性启发偏差**）。

**如何区分这两者？如何避免把可得性启发包装成 MVP 验证？**

这不是文字游戏——两者表面看一样（都是"应用新方法论"），但内核相反：

- MVP 验证的目的是**找到方法论的边界**
- 可得性启发**让你看不见边界**

# 张力

## MVP 验证立场

- 学新概念后不去应用 = 纯理论 = 不进认知（停留在"知道" vs "会用"）
- 应用是检验概念是否真理解的唯一方式
- 应用过程中的反馈 = 加深理解的核心机制
- 不应用 = 信息腐烂在 sys1 表层，永远内化不了

## 可得性启发立场（参考 [card-@认知偏差清单](card-@认知偏差清单.md)）

- "最近学的"在脑中权重最高 → 看见的所有问题都长得像它（**锤子-钉子现象**）
- 过度应用会让你**错过实际更适合的工具** —— 因为脑中只有那把锤子
- 越是漂亮的新理论，越容易被过度应用（**理论致盲**）
- 应用本身不是验证，**带反馈的应用**才是

## 两者都是"应用新方法论"——区别在反馈识别

|  | 健康 MVP 验证 | 可得性偏差驱动 |
|---|---|---|
| 出现 misfit signal | apply → misfit → **优先归因为工具不适配** | apply → misfit → 归因为"自己用得不好" |
| 后续动作 | 试别的方法 → 收边界 | 找借口让工具继续 → 通则化 |
| 元目标 | 找到工具的**边界** | 让工具成为**通则** |
| 时间尺度 | 较快收敛（N 次） | 延迟识别（N+M 次后才纠正） |

→ 关键不在"用了哪个工具"，在**出现 misfit signal 时的归因方向**。

# 不确定的地方（最重要）

- **如何在第一次应用时就识别 misfit signal？** 还是必须经过 N 次失败才能识别？
- 如果是"必须 N 次失败"，**N 的合理范围**是多少？2 次？5 次？10 次？
- "**诚实归因**"的标准是什么——什么时候归因工具，什么时候归因自己用得不好？
- 是否需要在 MVP 验证时**强制对比**另一种方法作为 baseline？
- **元认知本身可能失效**（参考 [card-@系统1系统2](card-@系统1系统2.md) §5）——"我现在是在 MVP 验证"这条判断本身可能就是可得性启发的产物
- 是否存在一个**通用的"早期 misfit signal"清单**？（候选信号："读起来跳" / "强行套用" / "效果勉强"）

# 候选 hypo（v2 实战推演）

源自 v2 §3-§4-§1 实战流程：

| 阶段 | 工具 | misfit signal | 当时归因 |
|---|---|---|---|
| 1. §3 用金字塔 | pyramid | 无 ✅ | "效果好" |
| 2. §2 用金字塔 | pyramid | 无 ✅ | "继续有效" |
| 3. §4 用金字塔 | pyramid | "读起来跳" ⚠️ | "§4 是元章 inherently 复杂"（**归因错位**） |
| 4. §1 必须用 matrix | matrix | "矩阵不适合 pyramid" | **金字塔有边界**（纠正） |

→ 这是一个延迟识别 misfit 的案例：signal 在 §4 出现但被错误归因，到 §1 才纠正——**延迟 = 1 个章节**。

## 临时 working model

1. 任何方法论的 MVP 验证时，设置**反例 quota** —— 主动尝试用 ≥1 种不同方法作为 baseline，建立对照
2. 完成 N 次应用后停下来 audit：是否有过 misfit signal？是否当时正确归因？
3. 把"通则"（硬性规则）→ "**默认 + 例外清单**"（v2 工作流软化已经实施了这个做法）
4. 出现 misfit signal 时，**优先归因为工具不适配**（默认归因方向）；只有当反例验证后才归因为"自己用得不好"

## 这个 hypo 的可疑之处

- **反例 quota 增加启动成本** —— 可能让 MVP 验证本身变得不"M"（不 minimal），违反"先跑通"的精神
- **诚实归因依赖元认知** —— 但元认知本身可能失效（自欺机制，[card-@系统1系统2](card-@系统1系统2.md) §5）
- **没给 N 的具体值** —— "N 次后停下来 audit" 在 N=∞ 时等于不 audit
- **延迟 1 个章节是否就算"失败"？** —— v2 的延迟最终被纠正，可能"健康延迟"本来就存在

所以这个 hypo 是**临时压缩的工作模型**，**不代表问题已解决**。
