# 笔记反思（note-reflect）

## 何时启用

- 用户明确要求复核：「检查一下」「这段逻辑通吗」「有没有幻觉」。
- **每次写完 atom-/hypo-/book-/ref- 的核心论证段后，主动跑一遍**，不需要等用户提醒。
- 用户在 atom-/hypo- 里抛出新假设时——这是最容易自欺的地方。

## 三层检查清单

按顺序走，**每一层都要给出"通过 / 存疑 / 失败"的结论**，不要含糊。

### 第 1 层：事实与出处（防幻觉）

- [ ] 引用的书名、作者、章节是否真实存在？**对照 [docs/](docs/) 下 book-@ 文件确认**，仓库里没有就明说"未在库内找到出处，需要用户核实"。
- [ ] 引用的数据/年份/人物是否凭空生成？数字默认存疑，除非有明确出处。
- [ ] 概念定义是否和库内已有定义一致？例如"系统 1/2"在 [book-@思考，快与慢](docs/book-@思考，快与慢.md) 里的定义是基准，新笔记里的用法不能漂移。
- [ ] 是否把"我推测的"和"作者说的"混成了同一句？必须显式区分。

### 第 2 层：逻辑结构（防断裂）

- [ ] **前提到结论中间是否跳步？**逐句问"这一句和上一句的关系是因果、并列、递进、对照中的哪一种"，说不清就是断裂。
- [ ] 是否有循环论证？（"X 是对的，因为 X 解释了 Y"——但 Y 的成立又依赖 X）。
- [ ] 反例是否被处理？如果只举正例，标注 `// 缺反例` 让用户决定。
- [ ] 条件依赖是否明示？"在 A 条件下 B 成立"和"B 总是成立"差别巨大。

### 第 3 层：分类与层级（防错位）

- [ ] 笔记前缀和内容匹配吗？讨论"我现在没想通的事"应该是 `atom-`，写成 `hypo-` 就是把问题伪装成答案。
- [ ] 标签层级正确吗？参考 [README.md](README.md) 第四节：
  - 认知层（思维结构、模型）
  - 行动层（执行方法）
  - 系统层（工作流、长期反馈）
- [ ] `coreConcepts` 是否过度概括？列出 5 个但其实只讲了 1 个，是常见的 AI 注水。

## 输出格式

反思结果**单独成块**，不要混在正文里：

```
## 自检（note-reflect）

- 事实层：通过 / 存疑：[具体哪句]
- 逻辑层：通过 / 存疑：[哪两句之间跳步]
- 分类层：通过 / 存疑：[前缀建议改成 X]

总体判断：可发布 / 建议作者本人复核 / 需要返工
```

## 反模式

- **不要默认"通过"**：如果三层全都标"通过"，说明你没认真检查。至少要主动找 1 个值得讨论的点。
- **不要替作者下定论**：自检的目的是**暴露不确定性**让作者决定，不是替作者判定真伪。
- **不要在自检里继续生成新内容**：发现问题就标出来，修改是另一步，由作者或下一轮对话决定。
